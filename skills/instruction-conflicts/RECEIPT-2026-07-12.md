# instruction-conflicts — forge receipt (2026-07-12)

Full cycle: headroom screen across the current codex-seat model family → forge on the model with
headroom → k-trial verify with a precision release gate → publish. All calls on the codex
subscription seat. Exact model ids pinned throughout; "codex" alone stopped being a sufficient
label the day the seat's family upgraded.

## 1. Headroom screen (k=1, 18 cases each, 0 backend errors)

| model | backend alias | score | failure mode |
|---|---|---:|---|
| gpt-5.5 (2026-07-11, historical) | codex | 15/18 | over-labels: spurious `output-format-conflict` in ~8 passes; 1 clean false-positive |
| gpt-5.6-sol | codex-56sol | 11/18 | under-detects: 5 of 7 misses answer `[]` on genuine conflicts; 4/4 cleans, near-exact answers |
| gpt-5.6-luna | codex-56luna | 15/18 | same profile as gpt-5.5 (identical 3 misses) |

Evidence: `out/live-instruction-conflicts/` (historical, hash-pinned in the published verification
record), `out/live-ic-56screen-sol/`, `out/live-ic-56screen-luna/`. The screen runs predate the
model-id-in-results change, so their cells carry backend aliases; the alias→id map is this
commit's `skills/instruction-conflicts/config.json`.

Finding worth the run by itself: the higher-intelligence family member (sol) is 4 cases WORSE at
this detection task than its cheap sibling — precision-perfect but conservative to the point of
waving real contradictions through. Score deltas between model generations are not monotone in
benchmark index; anyone swapping executor models without re-measuring their skills inherits this
silently.

Decision rule (pre-registered): target = sol if sol ≤ 16/18 → **forge target = gpt-5.6-sol**,
aim = recall without losing its precision.

## 2. Instrument failure, caught and fixed (first forge run invalidated)

The first forge run (10 contestants × 18 cases × k=3 = 540 calls) returned every contestant at
exactly 4/18 = the clean-case count. That pattern — identical scores, all-dirty-fail, zero
errors — is an instrument signature, not model behavior. Root cause: forge scoring prompts are
config-driven, and this skill had no `forge` block, so scoring fell back to a different task's
prompt shape. The model never saw the closed vocabulary; it answered with layer references like
`["global_user.3","project_guide.2"]` — often semantically CORRECT detections, scored as misses.
The generation prompt had the same wrong framing.

Fix: new generic forge mode `prompt-template` — contestants are scored through the skill's own
prompt template, byte-identical to the live runner's prompts for the original contestant. That
identity is now a regression test, and it doubles as a validity check on every future run: the
original contestant must reproduce the screen number (within k-noise) or the run is suspect.
Quarantined artifacts: `out/forge-ic-56-invalid-prompt-bug/` (kept, unpublished).

## 3. Forge run (valid instrument)

cases; 2 attempts × 4 mutation angles = 8 variants. Scoring model = `gpt-5.6-sol` (pinned in
results), k=3 strict-majority per case, 540 scoring calls, subscription seat.

**Strict lift: +27.8pp (61.1% → 88.9%), status = hero.** The variant spread (5/18 to 16/18)
shows generation-angle variance is real — most variants HURT; the search matters.

Per-case movement, original → v1: five under-detection misses fixed, each to a stable 3/3
trials (`ic-autonomy-edit-approval`, `ic-skip-tests`, `ic-runner-choice`, `ic-live-review`,
`ic-naming-checkin`). Still missed by everything ever tested here (5.5, luna, sol, any skill):
`ic-ambiguous-scorer` (the ambiguous-precedence category) and `ic-determinism` — the case set's
hard core.

What v1 actually changed (eyeball review of the diff): every conflict category gained one
positive example plus one near-miss NON-conflict; a mutual-exclusivity test replaced the vaguer
"disagree" ("not a conflict unless compliance with one directive prevents compliance with the
other"); directives are recorded with scope/conditions/strength before comparison. Its guards
push against over-labeling — the win is recall with precision held, not label spray. Generator
saw only the original SKILL.md; v1's examples derive from that file's own themes and none echo
the five fixed cases.

## 4. Precision release gate

The headline scorer passes a dirty case on ANY expected/got overlap, so a variant could buy
"lift" by spraying labels; only the 4 clean cases push back. Pre-registered gate (all at k=3, vs
the original on the target model): strictly higher subset score AND mean extra labels per dirty
case not higher AND clean passes not lower. Fail any → no lift declared regardless of headline.

**Verdict: ELIGIBLE** (`precision.py`, table in `precision.md` beside the results):

| | subset | exact-set rate | mean extra labels (dirty) | cleans (k-majority) |
|---|---:|---:|---:|---:|
| original | 11/18 | 0.389 | 0.310 | 4/4 |
| v1 | 16/18 | 0.556 | 0.310 | 4/4 |

Exact-set rate (not gated) improved too — v1 detects more while answering more exactly, the
opposite of the gaming profile the gate exists to catch.

Gate-tooling disclosure: the first gate computation returned INELIGIBLE from two defects in the
new `precision.py` — it read a result key forge cells don't carry (subset compared None), and it
demanded ALL k trials clean instead of the k-majority standard every other number in this
benchmark uses (one clean case passed 2/3 trials). Both fixed and regression-tested
(`test_clean_passes_use_runner_majority_not_all_trials`); the stricter all-trials view stays
visible as a diagnostic column (v1 = 3/4 all-trials clean vs original 4/4). The registered gate
standard, k-majority, is what the verdict above uses.

## 5. Adversarial judge (multi-model council on the diff)

Verdicts came back FAIL on the gate tooling with three findings adopted and fixed before
publish: (1) the precision gate failed OPEN on partial evidence — unparseable trials and unknown
case ids were silently dropped, so incomplete results could reach ELIGIBLE; now every case and
every trial must be present and parseable for both gated contestants or the tool refuses a
verdict; (2) exit codes failed open (0 on INELIGIBLE) — now 0/1/2 fail-closed for automation;
(3) the publish extensions (merged multi-run tables, appendix passthrough, model-id labels)
shipped without their own tests — now covered. One headline finding was dropped with cause: a
claimed TypeError crash misread `parse_array` (it returns a set; the gate demonstrably ran).
Re-run after hardening: ELIGIBLE, exit 0, full coverage (18 cases × 3 parseable trials for
original and v1).

## 6. Cross-model check (adoption guard)

A variant that lifts the strong model can regress a cheaper one (measured on this rig before:
a highsignal variant lifted GPT-5.5 +10.5pp and cost Haiku −10.5pp). Before recommending v1 as
the canonical skill, v1 and original were re-scored at k=3 on the other codex-seat models.

Instrument note: the first cross-model attempt was invalidated and quarantined
(`out/forge-ic-56-crossmodel-invalid-wrong-tree/`) — the runner script, executed by file path
from outside the repo, imported the installed package instead of the working tree (missing
aliases → 18/18 backend errors on one model; wrong prompt mode → the same wrong-task signature
as §2 on the other). Same seam family as script-vs-module import context; rerun executed with
the working tree pinned first on the import path, and the module provenance printed into the
run log. The wrong-task signature (§2) was recognizable within seconds the second time.

Adoption rule, registered before the rerun's numbers were seen: recommend v1 as the canonical
skill only if v1 ≥ original on BOTH other models (k=3 subset score); any regression → v1 ships
as a measured sol-specific variant, original stays canonical, tradeoff stated on the page.

Results (k=3 strict-majority, 0 backend errors in all four cells; module provenance printed in
the run log):

| model | original | v1 | delta |
|---|---:|---:|---:|
| gpt-5.6-sol (forge target) | 11/18 | 16/18 | +5 |
| gpt-5.6-luna | 14/18 | 14/18 | 0 |
| gpt-5.5 | 15/18 | 16/18 | +1 |

Validity echoes: original on gpt-5.5 at k=3 reproduces the published k=1 15/18 exactly; luna's
14/18 vs its k=1 screen 15/18 is a single borderline case, inside k-noise.

**Adoption verdict (per the registered rule): ADOPT.** v1 ≥ original on both other models — the
strong-model lift did not tax the cheaper tiers. v1 becomes the canonical SKILL.md (provenance:
generated blind by `gpt-5.6-sol`, selected by k=3 strict lift + precision gate, cross-model
checked); the pre-adoption original remains in git history at this receipt's commit and embedded
in the forge evidence. Adoption swaps the SKILL.md BODY only — the measured artifact; each home's
frontmatter (routing description, provenance links) stays, since frontmatter is stripped before
every scored prompt and was never part of the measurement.
